# Portfolio Version System — Design

**Date:** 2026-05-24
**Branch:** `feature/underwater-outro` (work stays on the current branch, per decision)
**Status:** Draft for review

## Goal

Turn the portfolio into a single Next.js app that hosts **multiple complete versions**
("projects") of the site, switchable at runtime. The **latest** version (declared in a
JSON manifest) is served at the root URL; older versions are browsable under `/v/<id>`.
No version's source code is lost — every version stays in the repo. Performance of the
live (latest) app is priority #1: archived versions must add zero runtime cost to it.

Two versions exist today:
- **2026** — the current Next.js app (`web/`). The latest; served at root.
- **2024** — the old hand-built static site (currently `src/`). Archived; served at `/v/2024`.

## Non-goals

- No Next-in-Next / Multi-Zones in this iteration (the 2024 site is static). The manifest
  reserves a `type: "zone"` so a *future* version that is itself a Next app can adopt
  Multi-Zones without a redesign.
- No release tooling (changelog/CI/tags). "Version" here means hosted, switchable
  snapshots + a build-info readout — not a release process.
- No relocation of the live `web/` app. It is root/latest; moving it is high-risk and
  serves no purpose.

## Decisions (locked with user)

| Topic | Decision |
|---|---|
| Hosting approach | **A** — static snapshot served via Next rewrites |
| Older-version URL | `/v/<id>` (e.g. `/v/2024`), outside the `[lang]` locale segment |
| Switcher / info UI | A section inside the renamed **Technical** panel |
| Manifest format | **JSON** in `web/src/data/` (matches existing `projects.json` etc.) |
| Contact redaction | Remove the contact section from the 2024 site **entirely** |
| Build readout | App version + git SHA + build date + **stack** (Next/React/Tailwind) |
| Settings rename | Full: label + i18n + route (`/settings`→`/technical` + redirect) + component |
| Branch | Stay on `feature/underwater-outro` |

## Repo layout

```
web/
  versions/2024/            ← 2024 site SOURCE (moved from /src), full code incl. .scss + maps. Committed. Frozen.
  public/v/2024/            ← runtime snapshot, generated from versions/2024 (excludes .scss/.map). GITIGNORED.
  src/data/versions.json    ← the version manifest
  src/...                   ← the 2026 app = latest, served at root (unchanged)
```

- Source of truth for 2024 = `web/versions/2024/` (in git → code never lost, no drift).
- The served copy is generated by a `prebuild` script (`web/package.json`) that copies
  `versions/2024/ → public/v/2024/`, excluding `*.scss` and `*.map`.
- Both dirs sit **under `web/`** so any deploy-root configuration (incl. Vercel rooted at
  `web/`) includes them.

> **Open layout question for review:** generated-and-gitignored `public/v/2024/` (single
> source of truth, needs the prebuild step to run) vs. committing the snapshot directly
> (no build step, but the frozen files live in git twice). Default = generated/gitignored.

## Components

### 1. Version manifest — `web/src/data/versions.json`

Ordered newest-first. Shape:

```jsonc
[
  { "id": "2026", "label": "Portfolio 2026", "year": 2026, "type": "next",   "path": "/",        "isLatest": true },
  { "id": "2024", "label": "Portfolio 2024", "year": 2024, "type": "static", "path": "/v/2024",  "isLatest": false }
]
```

- `type`: `"next"` (the live app) | `"static"` (snapshot served from `public/`) |
  `"zone"` (reserved for a future Multi-Zones Next app).
- Imported `import versions from "@/data/versions.json"`; a TS `Version` interface in the
  consuming module provides type safety (JSON stays plain).
- Single source consumed by both the router config awareness and the switcher UI.

### 2. Routing — `web/next.config.ts`

Add `async rewrites()` mapping the static snapshot:

```
/v/2024            → /v/2024/index.html
/v/2024/:path*     → /v/2024/:path*   (assets)
```

- No `middleware.ts` exists, and `/v/...` lives outside the `[lang]` segment, so locale
  routing is untouched and cannot swallow `/v/2024`.
- **Read `node_modules/next/dist/docs/` for the Next 16 rewrites API before writing this**
  (`web/AGENTS.md`: this Next 16 has breaking changes vs. training data).

### 3. 2024 site import + contact redaction

Move `/src` → `web/versions/2024/`, then in that source:

- **Redact contact (entirely):**
  - Delete the header nav `.contact` link (`<a id="headContact" href="mailto:…">`).
  - Delete the whole `#fix3` `.byeContact` block (`<h1>Contact me?</h1>` + email `<p>` +
    phone `<p>`). Phone `+41 78 775 39 78` and email `yannic.studer@protonmail.com` gone.
  - Remove the now-dead `#headContact` / `#fix3` references in `script.js`
    (`getElementById` + `classList.toggle` at lines ~45, 56, 63, 74) so nothing throws.
  - Apply the same redaction to `404.html` if it carries the contact markup.
- **Fix asset base:** inject `<base href="/v/2024/">` into `index.html` (and `404.html`)
  so the existing relative `./css`, `./js`, `./img`, `./svg`, `./manifest.json` paths
  resolve regardless of trailing slash.
- Also scrub the personal-contact phrasing from the `<meta description>`/OG/Twitter tags
  if they name contact info (they reference "contact information" generically — trim to
  taste; no PII there).

### 4. Technical panel (renamed from Settings) — `web/src/components/settings/`

- **Rename:** `dict.nav.settings` value → "Technical" (all locales: de, en); route
  `app/[lang]/settings` → `app/[lang]/technical`; component file `SettingsPanel.tsx` →
  `TechnicalPanel.tsx` (export renamed). Add a redirect `/[lang]/settings` →
  `/[lang]/technical` so existing links survive.
- **Versions section:** reads `versions.json`; lists each version (label, year) with a
  switch link (`path`), marks the one matching the current URL as active. Latest links to
  `/`, archived to `/v/<id>`.
- **Build section:** shows version + git short SHA + build date + stack versions, from
  build-time env (component 5). Added as a `Section` (the panel's existing card component)
  in the `diagnostics` tab, below the Hardware card. The Versions list also goes in
  `diagnostics`, above Build (it's read-only diagnostic info, same as the rest of that tab).

### 5. Build-time version/stack injection — `web/next.config.ts`

Capture at build and expose as `NEXT_PUBLIC_*` env (static, zero runtime cost):

- `version` ← `web/package.json` `version`.
- `gitSha` ← `git rev-parse --short HEAD` (fallback to a deploy env var, e.g. Vercel's
  `VERCEL_GIT_COMMIT_SHA`, when `.git` is absent).
- `buildDate` ← `new Date().toISOString()` at config eval.
- `stack` ← read `next` / `react` / `tailwindcss` versions from `package.json`
  (dependencies), surfaced as a small map.

## Data flow

```
versions.json ──► Versions section (switch links, active marker)
              └──► routing awareness (which path is latest/root)

next.config rewrites ──► /v/2024/* served from public/v/2024/ (static)

package.json + git ──(build)──► NEXT_PUBLIC_* ──► Build section readout
```

## Performance

- Live 2026 app: unchanged. No archived code enters its bundle or runtime.
- 2024 snapshot: plain static files, fetched only when a visitor opens `/v/2024`.
- Build readout: fully static, injected at build.

## Testing / verification

- `tsc` + `vitest` (the repo gates) stay green.
- `versions.json` parses and matches the `Version` type; `isLatest` is unique.
- Rewrite check: `/v/2024` returns the old site HTML; its CSS/JS/img load (relative
  paths via `<base>`); `script.js` runs without console errors (no missing-element throw).
- Redaction check: no `mailto:`, no `tel:`, no phone/email string anywhere under
  `web/versions/2024/` or `web/public/v/2024/`.
- Rename check: `/[lang]/technical` renders the panel; `/[lang]/settings` redirects to it.
- Build readout shows real version/SHA/date/stack.

## Out-of-scope follow-ups (noted, not built now)

- Snapshotting the 2026 app to `/v/2026` when a newer version becomes latest.
- Multi-Zones wiring for a future `type: "zone"` version.
- A standalone `/versions` archive page with previews.
